# Test 02 — 参数驱动：只给风格不给主题

## 用途

验证 Skill 能从**风格参数反推主题**。测试 `original_song_generator.md` 的冷启动决策表能不能工作。

## 输入

```
我要一首轻 R&B + 小调都市感 + 电钢铺底 + 副歌 Pad 扩张 + 城市夜景意象的歌。
```

## 预期行为

Skill 应当：

1. **识别出用户没给主题** —— 只给了 5 个风格参数，全部属于 system_prompt 的参数库
2. **两种合法反应之一**：
   - **反应 A**：主动用 1–2 个澄清问题问主题（例如"是讲独处、等待、还是告别？"），然后再出方案
   - **反应 B**：直接用默认映射反推主题（从"轻 R&B + 城市夜景" → 深夜独处 / 末班车 / 便利店凌晨），然后明确标注"已为你默认选定主题 X，如果不对请指定"
3. 两种反应都合格，但**不能完全自作主张**出一个毫无说明的主题

## 验收清单

### 必须项

- [ ] Skill 没有直接复用 examples.md 里的任何主题
- [ ] 输出要么是澄清问题，要么是带主题说明的完整 10 段
- [ ] 如果给了完整 10 段，主题与 5 个参数的风格气质对齐（不能给"毕业"这种错位主题）
- [ ] 参数都体现在最终方案里（轻 R&B 律动、小调、电钢、Pad、城市夜景，5 个都要能在输出里找到证据）

### 应当项

- [ ] 如果采用反应 A，澄清问题数量 ≤ 2
- [ ] 如果采用反应 B，主题选择有解释（一两句话说为什么选这个主题）
- [ ] 输出的情绪曲线符合小调都市感（内敛、不奔放、收束感明显）

### 禁止项

- [ ] 完全不问任何问题、也不说明默认主题 → FAIL
- [ ] 出一个违背所给参数的主题（例如"毕业典礼的白天"这种跟轻 R&B + 小调都市感完全冲突的主题）
- [ ] 参数列表里有一个被悄悄忽略（比如忘了 Pad）

## 手工评估点

- **主题合理性**：反推出的主题是否自然？还是生硬？
- **冷启动决策**：Skill 有没有展现 `song_analyzer.md §附录` 默认映射表的使用
- **参数一致性**：5 个参数在 10 段输出里是否互相协调，而不是强行拼接
