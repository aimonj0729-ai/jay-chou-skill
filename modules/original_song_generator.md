# Module E — Original Song Generator（编排器）

> 根据用户输入的完整创作需求，输出一首原创歌曲方案。
> 这个模块是**编排器**：调用 A（知识根）+ B（旋律）+ C（歌词）+ D（编曲），再经 F（自检）合成最终输出。

## 输入示例

> "写一首关于毕业前夜的华语流行歌，钢琴开头，情绪从克制到释怀，带一点轻 R&B 律动和东方画面感，副歌要有大记忆点。"

## 执行流程

```
1. 解析用户输入 → 提取参数
   - 主题（毕业前夜）
   - 情绪曲线（克制 → 释怀）
   - 风格参数（钢琴开头 / 轻 R&B 律动 / 东方画面感）
   - 硬性要求（副歌大记忆点）

2. 补全缺失参数（用 song_analyzer.md §附录的默认映射）
   - 毕业 → 慢板流行 + 大调抒情 + 青春回忆意象 + 钢琴主导 + 弦乐托举
   - 与用户指定合并 → tempo 80（兼顾慢板与轻 R&B）
   - → 小调轻 R&B 风格太重，调低，保持钢琴为主，R&B 只在副歌律动层

3. 加载词人人格（如指定）
   - 读 lyricist_persona.md，确定词人人格（fang-wenshan / huang-junlang / jay-self / auto）
   - auto 模式：根据主题 + 情绪 + 风格参数走自动匹配决策表
   - 词人人格影响步骤 4 中 lyric_builder 的所有输出维度

4. 加载融合风格（如指定）
   - 读 style_fusion.md，确定融合风格和比例
   - 按维度融合权重表分配：结构/情绪跟周杰伦侧，和声/节奏/编曲可融合
   - 检查兼容性矩阵，低兼容性时向用户确认

5. 查知识根 → 选择结构与和声方向
   - 钢琴抒情模型的默认结构（见 song_analyzer.md §3）
   - 大调抒情进行 + 副歌抬升
   - 融合模式下：按融合比例注入异质和声/结构元素

6. 分头生成
   - 调 melody_designer.md：每个段落的旋律设计（融合模式下注入融合侧旋律特征）
   - 调 lyric_builder.md + lyricist_persona.md：主题 + 意象库 + 示例句（按词人人格调色）
   - 调 arrangement_planner.md：六段式层次推进 + 东方点缀位置（融合模式下替换/混合音色）

7. 合成 10 段输出（对照 system_prompt.md 的模板）
   - 融合模式下：每个维度标注来源 [JC] / [F] / [MIX]
   - 融合模式下：额外输出"融合说明"段

8. 跑 similarity_guard
   - 命中警报 → 去相似化改写
   - 无警报 → 标 "PASS"
   - 融合模式下：额外检查融合侧是否引入了其他艺术家的标志性特征

9. 输出最终方案
```

## 10 段输出的内部生成顺序

按以下顺序生成，前一段的决策约束后一段：

0. **前置决策**：确定词人人格 + 融合风格（如有）
1. **核心概念**（先定，约束一切；融合模式下标注融合方案）
2. **情绪轨迹**（从核心概念推出，决定结构）
3. **结构建议**（小节数 + 段落顺序；融合模式下可引入异质结构元素）
4. **和声方向**（给定结构后，选和声家族；融合模式标注 [JC]/[F]/[MIX]）
5. **旋律设计建议**（给定结构 + 和声后，选音区 + 动机）
6. **歌词意象与叙事方案**（给定主题 + 结构 + 词人人格后，选意象系统和句法风格）
7. **编曲推进方案**（所有前面确定后，给层次推进；融合模式标注音色来源）
8. **副歌 hook 构思**（综合 5+6，给 hook 定位）
9. **原创性风险检查**（跑 similarity_guard）
10. **去相似化建议**（根据 9 的结果给改写）
11. **融合说明**（仅融合模式：哪些维度做了融合，来源标注汇总）

## 参数冲突解决规则

当用户给的参数互相冲突时，按以下优先级：

1. **主题 > 风格**：主题定情绪，风格只是手段
2. **情绪曲线 > 单点参数**：情绪连续性优先于局部参数
3. **用户显式要求 > 默认映射**
4. **结构合理性 > 参数叠加**：不硬塞所有参数，牺牲风格也要保结构

**示例冲突**：用户要"钢琴开头 + 轻 R&B 律动"
- 钢琴开头暗示慢板 / 温暖
- 轻 R&B 暗示都市 / 节奏感
- 解决：Intro 钢琴独奏 → V1 加 EP 和轻鼓 → 副歌 R&B 律动显现 → Bridge 回到钢琴
- 这样两个参数都被尊重，但不会在同一段互相打架

## 冷启动规则（用户给的信息很少时）

如果用户只给了 1–2 个参数（例如只给"毕业"或只给"钢琴"）：

- **不要立刻出完整方案**
- 从 `system_prompt.md` 的澄清问题清单里挑 1–3 个问
- 问完收敛后再进入上述流程

## 澄清问题选择优先级

当需要问澄清问题时：

1. 先问 **主题** 和 **情绪起点/终点**（决定整体方向）
2. 再问 **视角** 和 **段落功能偏好**（决定歌词方向）
3. 最后问 **tempo 和编曲核心**（决定技术参数）

每次最多问 3 个，不要一口气问完所有维度。

## 输出质量自检（在 similarity_guard 之前）

完整 10 段产出后，按下列清单做一次快速自检：

- [ ] 10 段都有，没有偷懒只给 5 段
- [ ] 段落之间有相互呼应（如结构和编曲对齐，和声和情绪曲线对齐）
- [ ] 副歌 hook 明确，不是"某种上行线条"这种模糊描述
- [ ] 歌词示例句有 3–5 句，都是原创
- [ ] 编曲方案有具体乐器，不是"钢琴进弦乐"这种没信息量的序列
- [ ] 情绪曲线的顶点在 C3 而不是 C1
- [ ] Outro 有明确的收束方式，不是"慢慢淡出"

不通过任一项 → 回到对应模块重写那一部分。
